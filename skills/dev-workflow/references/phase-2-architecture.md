# 阶段2：架构审查

## 执行步骤

1. **启动独立 sub-agent 执行架构审查**

   使用 `Agent` 工具：
   - `subagent_type`: `"general-purpose"`
   - `description`: `"独立架构审查"`
   - `prompt`:
   ```
   你是一个独立的架构审查专家。请审查设计文档（路径见下文），从指定维度给出评估报告。

   设计文档路径：<阶段1 产出的设计文档路径>

   审查维度（8个）：
   1. 模块划分 — 模块职责是否单一、边界是否清晰
   2. 接口设计 — 模块间通信方式是否合理、耦合度是否可控
   3. 数据流 — 数据流向是否清晰、是否存在循环依赖
   4. 可扩展性 — 新增功能是否可以不改核心代码
   5. 可测试性 — 每个模块是否可以独立测试
   6. 可移植性 — 平台相关代码是否隔离、抽象层是否合理
   7. 性能 — 是否存在明显的性能瓶颈
   8. 安全性 — 是否存在常见安全漏洞（注入、越权等）

   各维度评分标准（1-5星）：
   - 5星：无问题，可直接实施
   - 4星：有轻微问题，不影响实施
   - 3星：有中等问题，建议修改后实施
   - 2星：有严重问题，必须修改
   - 1星：设计存在根本性缺陷，建议重新设计
   每个评分必须附 1-2 句评分理由。

   【强制输出格式】必须严格按照以下结构输出，不要增删顶级章节：

   ## 架构审查报告
   ### 总体评价
   <2-3句>
   ### 各维度评分
   | 维度 | 评分(1-5) | 评分理由 |
   |------|----------|---------|
   | 模块划分 | X | <理由> |
   | 接口设计 | X | <理由> |
   | 数据流 | X | <理由> |
   | 可扩展性 | X | <理由> |
   | 可测试性 | X | <理由> |
   | 可移植性 | X | <理由> |
   | 性能 | X | <理由> |
   | 安全性 | X | <理由> |
   ### 发现的问题
   | 编号 | 问题描述 | 严重程度(严重/一般/建议) | 影响范围 |
   |------|---------|----------------------|---------|
   | P1 | <描述> | 严重 | <影响> |
   ### 改进建议
   <按优先级排序，每个建议附1-2句说明>
   ### 替代技术路线推荐
   <1-2条，每条含：适用场景、优劣对比、迁移成本评估>
   ### 结论
   <1-2句总结>

   不要将内容写入文件，输出完整报告文本即可。
   ```

   **如果 sub-agent 超时/失败：** 重试 1 次；仍失败则由主 agent 自行审查（降级）。

2. **主 agent 处理 sub-agent 输出**
   - 将 sub-agent 输出写入 `$REPORT_DIR/architecture-review-report.md`
   - 验证输出包含全部 7 个顶级章节
   - 验证发现的问题表有编号列
   - 向用户展示摘要

3. **替代路线评估**
   - 如果推荐了替代路线，询问用户是否采纳
   - 采纳 → 更新设计文档 → 重新审查（最多 2 轮）
   - 不采纳 → 继续

## 检查点

| # | 类型 | 验证内容 | 操作 |
|---|------|---------|------|
| 1 | 🔧 | 审查报告文件存在且含全部章节 | `Read` 前 80 行，检查 7 个章节标题 |
| 2 | 🔧 | 问题编号列表完整 | 检查「发现的问题」表格中有编号列 |
| 3 | 🔧 | 替代路线评估轮次 ≤2 | `Glob` 搜索 `$REPORT_DIR/architecture-review-report*.md` |

全部通过 → `git commit` → 更新 telemetry → 进入阶段3
